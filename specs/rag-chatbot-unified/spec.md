# Unified Feature Specification: Integrated RAG Chatbot

**Feature**: RAG Chatbot for Docusaurus
**Created**: Friday, March 13, 2026
**Status**: Draft
**Input**: User requirement to merge two RAG chatbot implementations into one, focusing on Google Gemini and Docusaurus integration.

## 1. Executive Summary
Develop and embed a Retrieval-Augmented Generation (RAG) chatbot into the existing Docusaurus documentation. The system will allow users to ask questions about the documentation and receive answers generated by Google Gemini, grounded in the book's content stored in Qdrant Cloud. It will also support context-specific queries on user-selected text, and store chat history in Neon Serverless Postgres.

## 2. Technology Stack
-   **Frontend**: Docusaurus (React/TypeScript)
-   **Backend**: FastAPI (Python).
-   **AI/LLM**: Google Gemini API (via `google-generativeai` / LangChain).
-   **Vector Database**: Qdrant Cloud (Free Tier).
-   **Relational Database**: Neon Serverless Postgres (for chat history/analytics).

## 3. User Scenarios & Testing (Mandatory)

### US-1: General Q&A (Priority: P1)
**As a** reader of the documentation,
**I want to** ask natural language questions via a chat widget,
**So that** I can find information without manually searching through pages.

**Why this priority**: This is the core functionality and provides immediate value to the user, enhancing their learning experience.
**Independent Test**: Can be fully tested by posing a question related to the book content and verifying the chatbot's response is relevant and accurate, and delivers immediate answers to reader queries.

**Acceptance Criteria:**
-   Chat widget is visible on all documentation pages.
-   User sends a query -> System retrieves relevant chunks from Qdrant -> Gemini generates answer.
-   Response cites or is based on the retrieved context.
-   If no relevant info is found, the bot politely declines.

### US-2: Chat with Selection (Priority: P1)
**As a** reader,
**I want to** highlight a specific paragraph or code block and ask a question about it,
**So that** I can get an explanation of complex concepts in context.

**Why this priority**: This refines the core functionality by providing a more granular and powerful way for users to interact with the content, addressing a specific explicit user requirement.
**Independent Test**: Can be fully tested by selecting text within the book, posing a question, and verifying the chatbot's response is accurate and restricted to the selected text, and delivers context-specific answers.

**Acceptance Criteria:**
-   User selects text -> A "Ask AI" tooltip/button appears OR the chat widget acknowledges the selection.
-   The selected text is sent as the primary context for the query.
-   The answer focuses specifically on the selected text.
-   If the question is not related to the selected text, the chatbot indicates it cannot answer outside the scope of the selected text.

## 4. Functional Requirements

-   **FR-001**: The RAG Chatbot MUST be embeddable within the published book (Docusaurus documentation).
-   **FR-002**: The chatbot MUST answer user questions accurately based on the content of the published book.
-   **FR-003**: The chatbot MUST be able to process and answer questions based *only* on user-selected text within the book.
-   **FR-004**: The chatbot MUST provide a mechanism for embedding new or updated book content into the vector database (re-indexing).
-   **FR-005**: The chatbot MUST use a vector database (Qdrant Cloud) for efficient retrieval of relevant book content.
-   **FR-006**: The chatbot MUST provide a user interface that is integrated seamlessly into the book's reading experience.
-   **FR-007**: The chatbot MUST indicate when it cannot answer a question due to lack of relevance in the book content or selected text.
-   **FR-008**: The chatbot MUST log chat interactions (user query, bot response) to Neon Serverless Postgres.

## 5. Architecture & Data Flow

### 5.1 Data Ingestion (Offline/Triggered)
1.  **Source**: `docs/` folder (Markdown files) of the Docusaurus project.
2.  **Process**:
    -   Load Markdown files.
    -   Split content into manageable chunks (e.g., using `RecursiveCharacterTextSplitter` from Langchain).
    -   Generate Embeddings using Gemini's `models/text-embedding-004` (or compatible model).
    -   Upsert generated embeddings and corresponding text chunks to Qdrant Cloud Collection (`robotics_textbook`).
    -   Mechanism to trigger re-indexing when book content is updated.

### 5.2 Query Flow (Runtime)
1.  **User** asks question in Docusaurus UI (via the chat widget or "Ask AI" selection).
2.  **Frontend** sends request (`query: str, selection: str (optional), session_id: str`) to **FastAPI Backend** (`POST /chat` endpoint).
3.  **Backend**:
    -   Generates embedding for the incoming query (and optional selection).
    -   Searches Qdrant Cloud for nearest vectors (Top K relevant chunks) based on the query embedding.
    -   Constructs a comprehensive prompt for Gemini: User Query + Retrieved Context + (Optional) Chat History from Neon.
    -   Calls **Gemini API** for completion to generate the chatbot's response.
    -   Logs the interaction (`session_id, user_query, bot_response, timestamp`) to **Neon Postgres**.
4.  **Backend** streams or returns the generated response to the **Frontend**.

## 6. Implementation Requirements

### 6.1 Backend (FastAPI - `rag/server.py`)
-   Implement `POST /chat` endpoint: Accepts `{query: str, selection: str (optional), session_id: str}`. Handles RAG logic, Gemini API call, and Neon logging.
-   Implement `POST /ingest`: Admin endpoint to trigger re-indexing (MUST be protected, e.g., API key, authentication).
-   Establish database connection to Neon for storing chat history (`session_id, user_query, bot_response, timestamp`).
-   Include CORS middleware to allow requests from the Docusaurus frontend.

### 6.2 Vector DB (Qdrant Cloud)
-   Dedicated Collection: `robotics_textbook`.
-   Vector Size: 768 (standard for Gemini embeddings).
-   Distance Metric: Cosine.

### 6.3 Frontend (Docusaurus - `src/components/ChatWidget.js`)
-   Develop a custom React Component named `ChatWidget`.
-   Integrate `ChatWidget` into the Docusaurus site's root layout or via `swizzle` for global availability.
-   Implement user interface for chat: floating button, chat window, input field, message display.
-   Implement logic to detect user text selection and present an "Ask AI" option, sending the selected text as context.
-   Manage chat history within the component (local storage/session-based for display, server-side for persistence).

## 7. Edge Cases
-   **Content Updates**: The RAG system (via `/ingest` endpoint) should be able to re-index updated book content to maintain accuracy.
-   **Input Length**: Gracefully handle very long user questions or selected text, potentially through truncation or summarization before sending to embedding/LLM.
-   **No Information Found**: If no relevant information is found in the book for a query, the chatbot should clearly inform the user that it cannot find an answer.
-   **API Limits/Errors**: Implement robust error handling and fallback mechanisms for external API calls (Gemini, Qdrant, Neon).

## 8. Success Criteria (Measurable Outcomes)

-   **SC-001**: 90% of user questions about the book's content receive accurate and relevant answers.
-   **SC-002**: 95% of questions based on user-selected text receive accurate and relevant answers restricted to the selected context.
-   **SC-003**: The chatbot response time for a typical query is under 3 seconds.
-   **SC-004**: The chatbot can be easily integrated into the Docusaurus build process.
-   **SC-005**: Users report a 75% satisfaction rate with the chatbot's ability to answer questions.
